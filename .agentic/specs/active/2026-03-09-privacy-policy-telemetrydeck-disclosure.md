# Privacy Policy TelemetryDeck Disclosure Spec

Status: Approved
Date: 2026-03-09
Owner: Codex

## Problem Statement
The current privacy policy uses generic analytics language and does not clearly disclose the mobile analytics integration with TelemetryDeck, what data categories are collected, what is not collected for this analytics flow, why analytics data is processed, how long it is retained, and what user control/contact path exists.

## In Scope
- Revise analytics/privacy sections in the public privacy policy to explicitly name TelemetryDeck.
- Add analytics data category details:
  - anonymized user identifier,
  - event names,
  - coarse device/app metadata,
  - timestamps.
- Add a statement that app analytics flow does not use cookies and does not rely on storing IP addresses for analytics tracking.
- Add purpose and legal basis language for analytics processing.
- Add analytics retention wording aligned to TelemetryDeck default retention behavior.
- Add a user controls/contact path for analytics questions and requests.
- Update “Last updated” date in the privacy policy.

## Out of Scope
- Changes to app telemetry implementation code.
- Cookie banner/consent UI implementation.
- Terms page changes.
- Jurisdiction-specific legal rewrite beyond scoped disclosure additions.
- Any non-privacy page updates.

## Constraints
- Keep page styling and structure consistent with existing legal pages.
- No new dependencies.
- Limit code/content changes to files listed in File Impact Plan.
- Do not make legal claims that conflict with actual analytics behavior.

## Acceptance Criteria
1. Privacy policy explicitly names TelemetryDeck as the analytics provider.
2. Policy includes all requested analytics data categories (anonymized user identifier, event names, coarse device/app metadata, timestamps).
3. Policy explicitly states analytics flow does not use cookies and does not use stored IP addresses for app analytics tracking.
4. Policy includes analytics purpose and legal basis text.
5. Policy includes explicit analytics retention wording (provider-default retention behavior).
6. Policy includes a clear contact/user-controls path for analytics questions.
7. “Last updated” date reflects this revision.
8. Build succeeds after content update.
